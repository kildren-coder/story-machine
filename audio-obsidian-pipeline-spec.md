# 音频知识图谱工作流 — 项目说明文档

> 本文档面向接手本项目的 Agent（如 Claude Code），目的是让你在不重新和用户对齐一遍的情况下，快速了解项目背景、已确定的设计决策、和尚待实现的部分。请将本文档视为唯一可信的需求来源；如与用户临时口头描述冲突，以本文档记录的设计原则为准，除非用户明确要求变更。

## 1. 背景与目标

用户日常需要处理大量时长约 3 小时的音频，内容以政经类为主：有的是时事评论，有的是跨国、跨百年的宏观历史脉络分析。这类内容对细节保真度要求很高，且人物、事件、群众心理、政治动态彼此高度关联。用户听不过来，希望用 AI 工作流解决两个需求：

1. **精确转写**：忠实反映音频原文，不能走样、不能被模型脑补。
2. **自动组织进 Obsidian**：把内容持续沉淀为一张人物/事件/概念/历史时期彼此关联的知识图谱，而不是互不相干的一堆笔记碎片。

用户明确希望流程中保留人工介入环节——TA本身也需要对内容做主观理解，不是要一个纯黑箱的全自动系统。

## 2. 硬件与账号现状

- **笔记本电脑**：用户的主力设备，日常开发（包括另一个 AFK 项目）和 Obsidian 库都在这台机器上，是绝大部分时间使用的设备。
- **房间内 PC**：配备 **RTX 5070**（12GB 显存），平时可能处于睡眠/关机状态，专门用来跑本地语音识别模型。
- **Claude 订阅**：目前是 Pro，可能升级到 **Max 5x**（视额度实际消耗决定）。Claude 和 Claude Code 共享同一额度池，采用 5 小时滚动窗口 + 每周上限的双层限额。
- **Gemini**：计划使用 Google AI Studio 的 **Gemini Flash 免费 API 额度**做批量机械性提取，以节省 Claude 订阅额度。
- **NotebookLM**：免费版，仅作为用户个人理解内容的并行工具，**不接入自动化主链**（无 API，无法程序化提取内容）。

## 3. 总体架构

```
[笔记本：Obsidian + Claude Code]
         │   (Tailscale + SSH 桥接)
         ▼
[房间PC：RTX 5070 + faster-whisper]
```

- 用 **Tailscale** 建立两台机器间的固定虚拟内网，避免路由器端口转发和 IP 漂移问题，笔记本离家在外也能连上 PC。
- PC 上开 **SSH 服务**（如 Windows OpenSSH Server），配置密钥免密登录。
- 建议给 PC 配置 **Wake-on-LAN**，笔记本可远程唤醒 PC 后再派发任务。
- 需要一个笔记本本地的包装脚本：上传音频到 PC（`scp`/`rsync`）→ 远程触发转写（`ssh` 执行 whisper 命令）→ 取回逐字稿（`scp`）。理想情况下应封装成 CLI 的一个子命令，可被 Claude Code 直接调用。
- ASR 之后的所有环节（分块、提取、人工核对、合并查证、写入 Obsidian）都在笔记本本地完成，不再依赖 PC。

## 4. 完整处理流程

### 阶段 0：音频转写（在 PC 上执行，经桥接触发）

- 工具：`faster-whisper`（或 `whisper.cpp`），模型用 `large-v3`。配置已在 #11 照真机实测固化：`float16` / `batch_size=16` / VAD 开 / `word_timestamps=True` / `language="zh"`（实测 43 倍实时）。#24 收尾实测补证：`zh` 强制与 auto 检测全篇输出逐字节一致，英文短语在 `zh` 下完整存活、切换处词时间戳可靠，`multilingual` 无增益不启用；VAD 关闭在音乐段大规模幻觉（且批处理模式架构上必须 VAD）——`language="zh"` + VAD 开是终局配置，不再复议。
- RTX 5070 是 Blackwell 架构，需要较新的 CUDA（12.8+）及对应 PyTorch/CTranslate2 构建版本，环境搭建报错时优先排查这一点。
- **说话人分离子步**（#11/#15 拍板）：转写后同机顺序跑 3D-Speaker（CN-EN CAM++），纯 CPU、独立 venv、不开重叠检测；K 默认自动估计上限 4，CLI 可 `--speakers N` 锁死；贴合脚本用词时间戳在说话人切换点拆段后挂 `speaker` 标签。禁用 `pyannote.audio` 4.x（显存尖峰 bug）。
- 输出（#11 拍板）：**JSON 正本**——段级 `{start, end, speaker, text}`，浮点秒，匿名 `SPEAKER_XX`，永久不可变；`EP{n}.txt` 为派生渲染 `[HH:MM:SS] SPEAKER_00: ……`。中英混杂内容保留原文语言，不强制音译。匿名→身份映射每集一次人工点名，落 `EP{n}.meta.json`，转写后、提取前填（声纹自动化 → #27）。
- **热词库机制**（#11/#24 拍板修订；解决用户不认识欧美冷门实体、无法手写热词表的问题）：
  1. `initial_prompt` **封死**（#24 实测 prompt 泄漏幻觉，撞"ASR 不得脑补"红线），改用 faster-whisper 的 `hotwords` 参数。
  2. 持久 `hotwords.json` 只收**多字词**（专名/成语；成语通用表一次性导入）；短字母缩写不收录（解码期救不动，归修复 pass 的语境规则）。
  3. 提取前跑一道 **LLM 修复 pass**（Gemini Flash，只修同音词、禁增删）：按阶段 1 的 20-30 分钟块无重叠、顺序处理；出口设机械校验闸门（输出同构：段数/时间戳/speaker 不变，逐段编辑距离超 ~15% 则回退原文）；每块附带产出新专名清单 → 集内滚动词表 `EP{n}.glossary.json` 随后续块传递；差分文件 `EP{n}.repairs.json` 供抽查。修复稿为派生文件，原始正本永久保留。
  4. 词表入口只有人工回流：阶段 3 审核时把确认的新词/语境规则补进 `hotwords.json`（滚动词表不自动入库）。
  5. 热词库是长期资产，随处理集数增加而越来越完善，**不要每次重新生成**；与别名库共用一份数据（别名库是热词库种子）。
  6. **硬边界**：`hotwords` 注入有 **223 token 静默截断上限**（faster-whisper `transcribe.py` 源码实证：`max_length // 2 - 1`，超出部分无告警丢弃；与 `initial_prompt` 共享同一 prompt 窗口，#18 报告结论迁移验证）。脚本注入前应计 token 数、超限时告警；库攒大后"按集选词"是运营问题，实跑逼近上限再定策略。

### 阶段 1：分块（Map-Reduce 的 Map 准备）

- 逐字稿按 **20–30 分钟**切块，块间保留 **1–2 分钟重叠**，避免跨块的实体关系被切断，尽量在句子边界切。
- 目的：3 小时逐字稿约 4-6 万字（60-90k tokens），不应一次性整体处理——分块后单块 token 量在 1-2 万左右，能保持模型对细节的注意力，并控制在用户要求的 **120k 上下文（"smart zone"）**以内。

### 阶段 2：结构化提取草稿（Gemini Flash，免费额度）

对每个分块调用 Gemini Flash，输出结构化草稿（JSON 或统一格式 Markdown），包含：

- 涉及的人物/事件/国家地区/时期，及一句话描述
- 实体间关系（因果、影响、时序等）
- **每条论断打标签**：`史实类` / `时事断言类` / `观点类`（观点类不进入查证队列）
- 每条内容标注来源时间戳

草稿写入 Obsidian vault 的暂存区，**不加双链**：

```
Vault/_review/EP{编号}_draft.md
```

### 阶段 3：人工核对（人工介入的核心环节，不可省略）

用户在 Obsidian 中直接打开草稿文件校对：

- 修正错误识别的实体名/别名
- 删除明显无意义或幻觉性的提取结果
- 对存疑判断打标记（如 `?`），供查证阶段重点处理

这一步同时是质量闸门：防止 Gemini Flash 的提取错误污染正式笔记网络。确认无误后需要一个明确信号触发下一阶段（具体机制见第 9 节待确认事项）。

**校对配对数据（廉价保险，不可省略）**：用户的校对动作本身就是高质量标注——「模型草稿 → 人工修正后」的配对，是为本项目 schema 与题材量身定做的微调数据集，且只能在校对当下被捕获，事后无法补做。

- 阶段 2 写出草稿时，**同时写一份只读快照**：`_pairs/EP{n}/chunk{k}_pre.md`
- 阶段 3 确认信号触发时，把用户校对后的版本存为 `_pairs/EP{n}/chunk{k}_post.md`
- 两份文件均为永久资产，与逐字稿同级
- MVP **不对这批数据做任何加工**（不做 diff、不做统计、不做训练）——只负责无损保存

用途：本地模型微调（若阶段 2 引擎走本地路线，这是唯一入场券）、提示词迭代的回归测试集、提取质量随时间的变化度量。成本是多写一个文件，因此无论最终选哪个提取引擎都应保留。

### 阶段 4：合并、消歧与事实核查（Claude Code）

1. **实体匹配与合并**：读取 `_index/entities.json`（记录标准名 + 全部别名），对草稿实体做精确匹配（含别名）+ 模糊消歧判断。已有实体 → 在其笔记"来源记录"追加内容并注明集数/时间戳；新实体 → 按模板新建笔记。
2. **事实核查**：仅针对 `史实类` 和 `时事断言类`，优先处理用户标记的疑点。联网搜索核实，优先原始信源（政府文件、当事媒体、学术资料），笔记中附判定结果（属实/部分属实/未找到支持/存在争议）+ 依据链接。**每集查证条数需设上限**（如 10-20 条），这是整条流水线里额度消耗最大的环节。
3. **生成/更新正式笔记**：人物/事件/概念笔记按模板写入正式目录，同时生成本集单集笔记（MOC），链接到当集所有涉及实体。

### 阶段 5：落库与图谱呈现

最终笔记全部用 `[[双链]]`，Obsidian Graph View 自动呈现关联网络，无需手动连线。

**并行分支（不进入以上数据流）**：逐字稿到手后，用户可自行上传至 NotebookLM 做个人理解与提问，与自动化流程互不干扰，无需集成。

## 5. Obsidian Vault 结构与模板

```
Vault/
├── 10-Episodes/         单集笔记（MOC）
├── 20-People/           人物
├── 21-Events/           事件
├── 22-Places/           国家/地区
├── 23-Concepts/         概念/思潮
├── 24-Periods/          历史时期
├── _review/             人工核对暂存区（草稿，不参与图谱）
├── _pairs/              校对配对数据（pre/post 快照，永久资产，不参与图谱）
└── _index/entities.json 实体索引（程序读写，非面向人）
```

实体笔记模板（以人物为例）：

```markdown
---
type: person
aliases: [别名1, 别名2, 英文原名]
tags: [国家, 相关主题]
---
# {{标准名}}

## 概述
（首次创建时写一句话定位）

## 相关笔记
- [[相关事件]]、[[相关概念]]

## 来源记录
- [[EP{n} 标题]] {时间戳} —— 该集提到的具体内容摘要
```

单集笔记模板包含：整集摘要、按时间戳排列的要点、本集涉及实体清单（全部双链）、查证记录小节。

## 6. 模型/工具分工与成本策略

| 环节 | 模型/工具 | 计费方式 |
|---|---|---|
| 语音转写 | 本地 faster-whisper large-v3（PC/5070） | 免费（电费） |
| 逐字稿纠错、热词确认 | Gemini Flash 或本地小模型 | 免费额度 |
| 分块结构化提取（Map） | Gemini Flash | 免费额度 |
| 实体消歧、合并、笔记生成（Reduce） | Claude Code（Sonnet 级别，疑难可用 Opus） | Claude 订阅额度 |
| 新闻/时事类事实核查 | Claude Code（需联网搜索） | Claude 订阅额度，消耗最大 |
| 个人理解与提问（并行，非主链） | NotebookLM | 免费 |

- `anthropic` API 直接调用与 Claude Code 订阅是两套独立计费体系，Pro/Max 订阅额度**不能抵扣 API credits**。
- 用户同一账号还处理另一个 AFK 开发项目，与本项目共享额度池，建议流水线按"每天 1-2 集"节奏处理，避免集中爆发撞上每周上限。
- 若 Pro 持续不够用，优先试 Max 5x（$100/月），观察两周实际消耗（`/usage` 查询余量）后再决定是否上 Max 20x。
- 查证深度是最大的可调成本旋钮，额度紧张时优先收紧此项。

## 7. 关键设计原则（务必遵守）

1. **转写与整理严格分离**：ASR 阶段只做听写，不允许模型顺带总结/润色/脑补；原始逐字稿 + 时间戳是唯一事实基础。
2. **提取阶段不得脑补**：提示词必须明确要求"只提取逐字稿中明确出现的内容，不要补充模型自身的背景知识"。
3. **Map-Reduce 而非一次性长文本**：超过约 2 万 token 的逐字稿必须先分块独立提取，再对提取结果（而非原文）做归并。
4. **同一实体必须用同一名字**：靠 `aliases` 字段和实体索引文件防止重复建档，这是自动织网能否成立的关键。
5. **人工核对是质量闸门，不是可选项**：草稿必须先进暂存区等用户确认，才能进入正式笔记网络。
6. **查证与观点分离**：只对 `史实类`/`时事断言类` 查证，`观点类` 只归档不查证；查证结果必须附来源链接。

## 8. 建议的开发优先级

1. 打通笔记本↔PC 桥接（Tailscale+SSH）与远程转写脚本，先手动跑通"一句命令，音频进逐字稿出"
2. 实现分块逻辑 + Gemini Flash 提取草稿（先单块，再批量）
3. 搭建 Obsidian 目录结构与笔记模板，实现 `_review/` 暂存区读写
4. 实现实体索引（`entities.json`）与匹配/消歧逻辑
5. 接入 Claude Code 的合并、查证、笔记生成逻辑
6. 打磨热词库机制，让转写准确率随处理集数递增
7. 视实际使用情况调整分块粒度、查证条数上限等参数

## 9. 待确认/开放问题

- 分块重叠时长、每集查证条数上限等参数的具体默认值——建议先用保守值跑通几集再调优。
- `_review/` 草稿"确认通过"具体用什么信号触发下一阶段（改文件名/加 frontmatter 字段/CLI 命令行参数皆可），需要与用户约定一个简单可靠的机制。
- 是否需要给 CLI 加"批处理队列"模式，自动按每天 1-2 集节奏处理积压音频，还是完全手动逐次触发。
- Max 5x 是否够用需要实跑两周观察后再决定是否升级 Max 20x。
