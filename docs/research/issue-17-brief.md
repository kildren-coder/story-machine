# PR 摘要：issue #17 — Gigafact Parser 架构参考

## 问的是什么
闭源新闻室工具 Gigafact Parser 怎么把"断言"绑到"谁说的"、有无置信度/校验机制、吞吐量多少——给 #13 归因层、#15 说话人层当架构参照。

## 答案是什么
Parser 落地的正是"claim→发言人→数据库规范 profile→时间戳回链"四元绑定，与我们方向同构；质量保证靠人工在环＋可覆盖＋暴露推理＋回链原录音，无公开数值置信度；机器"一小时音频→分钟级"，人工端到端约压缩 50×（仅"发现 claim"、不含查证）。准确率/模型/成本一律查不到。

## 对项目意味着什么
确认"抽取≠查证"分层与现有绑定架构合理，无需改。#13 宜做成 provenance-first（每条归因回链逐字稿时间戳）；"置信度数值必填"是比 Parser 更严的加法，阈值无外部锚点、只能自决。#15 须给发言人判定留人工覆盖位。

## 最不可靠的地方
① §5"底层建在 OpenAI 栈"仅凭搜索摘要、未验一手——错了会误导技术选型；验证：查 Gigafact/OpenAI 一手公告再引用。② 吞吐数字全出自 partner（Wisconsin Watch）自述、非独立评测，恐偏乐观；验证：找第三方独立评测复核后再当基线。
