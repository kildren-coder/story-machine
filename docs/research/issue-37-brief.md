## 问的是什么

阶段 3 人工审完 `_review/EP{n}_draft.md` 后，用什么信号触发阶段 4？四种候选——frontmatter 字段 / 文件名 / 目录移动 / 显式 CLI 命令——在**误触发抵抗力（首要）**、部分完成表达、幂等重放三维度上，现成项目（Decap CMS / Hugo / Jekyll / paperless-ngx / Obsidian 原生 / 标注工具）怎么做的？

## 答案是什么

**抄 Decap 的双层结构：状态放好改的地方，推进靠一个显式动作。**

- **人面信号 = frontmatter 布尔** `reviewed: true`（Obsidian 原生 checkbox property，移动端可点、无双链副作用、Bases 可筛）。草稿模板播默认 `reviewed: false`——Hugo 教训：安全默认是「未通过」，通过必须显式翻转。
- **推进 = 一条显式 CLI 命令**（形状归 #5）。人存盘后手动敲：①读 `reviewed:true` 当前置条件否则拒推（双闸门）；②即刻读静止文件写 `chunk{k}_post.md` 快照（无监听竞态）；③把草稿移出 `_review/`（幂等，学 paperless「消费即移出」）。
- **部分完成**（审了 3 块还剩 3 块）用块级 `- [ ]`/`- [x]` checklist 承载，独立于整集 `reviewed`（Label Studio/Argilla 教训：半成品是一等状态）。

**排除**：目录监听/改名当触发器——Obsidian 写盘是否原子官方无保证（一则截断报告被开发者反驳但未证伪）、改名重写双链、移动端同步把移动当「删+建」易冲突。

## 对项目意味着什么

给 spec 第 9 节开放问题一个可拍板方案：frontmatter `reviewed` + 显式 CLI 命令双层。阶段 3 补两条约定——快照由确认命令读静止文件（不走监听）、块级 checklist 表达部分完成。与 #5 边界清晰：本票只定信号形态，命令参数/续跑归 #5。

## 最不可靠的地方

① Obsidian 写盘原子性——只能确认「官方无文档保证 + 一则截断报告被反驳但未证伪」，两个方向都悬着；推荐「CLI 触发不监听」正是为绕开这个未知，若未来真要监听需真机实测。② 块↔集文件粒度：spec 草稿是每集一文件、`_pairs` 是每块一文件，二者如何对应影响 checklist 落地，已列未决问题留 #4/#5。③ 移动端无终端敲 CLI——若用户真有「手机审完即推进」诉求，需旁路，会重新引入竞态权衡。
